# Урок 10. Финальная сборка: архитектура, интеграция и защита проекта

Продолжительность: 2 часа 30 минут.

## Результат урока

Студенты соберут все изученные идеи в законченное приложение, проследят полный поток данных, проведут проверку доступности и состояний, выполнят небольшое самостоятельное расширение и защитят архитектурные решения.

## Сценарий преподавателя

### 0–15 минут. Мы уже построили продукт

Запустите приложение и без кода пройдите пользовательский сценарий:

1. найти фильм;
2. выбрать жанр;
3. отсортировать;
4. добавить в избранное;
5. открыть детали;
6. обновить страницу;
7. выполнить поиск через сеть, если доступен ключ.

Скажите:

> Финальный урок не про магический новый хук. Он про то, как небольшие идеи работают вместе. Настоящая сложность появляется на стыках функций.

Попросите студентов назвать урок, на котором появилась каждая возможность.

### 15–40 минут. Архитектура компонентов

Откройте папку `src/components`.

Нарисуйте:

```text
App
├── Navbar
├── SearchPanel
├── MovieCard × N
│   └── Icon
├── EmptyState
└── MovieModal
    └── Icon
```

> `App` координирует данные и состояния. Компоненты представления получают значения и callback-функции. Сервис общается с API. Файл данных хранит локальный каталог.

Разберите ответственность файлов:

- `Navbar` — навигация и режим избранного;
- `SearchPanel` — элементы формы;
- `MovieCard` — краткое представление фильма;
- `MovieModal` — детали и жизненный цикл диалога;
- `EmptyState` — восстановление из пустого результата;
- `Icon` — переиспользуемая графика;
- `services/omdb.js` — внешняя граница.

> Хорошая архитектура не означает максимальное количество файлов. Она означает, что изменение имеет очевидное место.

Интересный факт:

> В больших приложениях компоненты часто организуют по функциям продукта, а не по типам файлов. Нет единственной идеальной структуры — важны ясные границы и согласованность команды.

### 40–65 минут. Карта состояния

Вместе составьте таблицу:

| Состояние | Что описывает |
| --- | --- |
| `catalog` | текущий источник фильмов |
| `query` | текст поиска |
| `genre` | выбранный жанр |
| `sortBy` | порядок списка |
| `favoriteIds` | сохранённые ID |
| `favoritesOnly` | режим показа |
| `selectedMovie` | открытый фильм или `null` |
| `status` | этап сетевого запроса |
| `errorMessage` | понятная ошибка |
| `sourceLabel` | подпись источника данных |

Спросите:

> Какие значения не являются состоянием?

Разберите `genres` и `visibleMovies` как производные данные в `useMemo`.

Проследите один полный сценарий:

```text
клик по сердцу
→ MovieCard вызывает onToggleFavorite(id)
→ App обновляет favoriteIds
→ useEffect сохраняет IDs
→ App рендерится
→ Navbar получает новый count
→ карточка и модальное окно получают новый isFavorite
```

> Один источник истины автоматически синхронизирует несколько частей интерфейса.

### 65–75 минут. Перерыв

Перед перерывом попросите нарисовать поток поиска через сеть без подсказки.

### 75–100 минут. Интеграция и крайние случаи

Проверьте сложные комбинации:

- фильтр плюс избранное;
- сортировка результатов API без рейтинга;
- открытие фильма и переключение избранного внутри modal;
- сброс после пустого результата;
- ошибка сети после уже показанного каталога;
- поиск через сеть, когда включён режим избранного.

Разберите:

```js
(b.rating ?? 0) - (a.rating ?? 0)
```

> Результаты поиска могут не иметь рейтинга. Nullish coalescing даёт безопасное число только для `null` или `undefined`.

Покажите `resetCollection`.

> Сброс — важная продуктовая операция. Он должен вернуть связанные состояния в согласованную исходную точку, а не очистить только одно поле.

Обсудите возможное улучшение:

> Сейчас `query` одновременно служит локальным фильтром и текстом для отправки API. В более крупном продукте можно разделить `inputQuery` и `submittedQuery`, чтобы ввод после запроса не фильтровал серверные результаты неожиданно.

### 100–120 минут. Дизайн-система и качество

Откройте `styles.css` и несколько компонентов.

> Четыре цветовых токена и один display-шрифт создают общий визуальный язык. Utility-классы описывают конкретную композицию каждого компонента.

Попросите найти примеры:

- responsive variants;
- hover и focus;
- disabled;
- arbitrary values;
- grid и flex;
- условные классы;
- `sr-only`;
- `aria-live`, `role="alert"`, `aria-pressed`;
- возврат фокуса из modal.

Проведите короткую проверку:

1. клавиатура;
2. ширина 375 пикселей;
3. консоль без ошибок;
4. Network offline;
5. обновление с избранным;
6. пустой результат.

Интересный факт:

> Качество интерфейса проявляется не только в успешном сценарии. Загрузка, пустой результат, ошибка, клавиатурное управление и медленная сеть — такие же реальные состояния продукта.

### 120–140 минут. Финальное практическое испытание

Разделите студентов на пары. Каждая пара выбирает одно расширение:

1. кнопка очистки всех фильтров;
2. сортировка по алфавиту;
3. счётчик активных фильтров;
4. отмена устаревшего API-запроса;
5. список недавно открытых фильмов;
6. отдельный компонент сообщения об ошибке;
7. русификация пользовательского интерфейса;
8. тест функции фильтрации после её выделения из компонента.

Условия:

- сначала назвать владельца нового состояния;
- не мутировать существующие массивы;
- добавить loading/empty/error, если функция связана с сетью;
- проверить клавиатуру;
- объяснить изменение за две минуты.

Ходите между парами и задавайте вопросы:

> Это новое состояние или производное значение? Кто должен им владеть? Что произойдёт при пустых данных? Как пользователь вернётся назад?

### 140–150 минут. Защита и ретроспектива

Каждая пара коротко показывает:

- что добавила;
- какие файлы изменила;
- где живёт состояние;
- какой крайний случай проверила.

Финальные вопросы всему классу:

1. Как данные попадают из API в карточку?
2. Почему избранное хранит ID?
3. Когда нужен эффект?
4. Какие части можно вычислить без состояния?
5. Что делает modal доступнее?
6. Как Tailwind узнаёт классы проекта?
7. Какой следующий рефакторинг вы бы сделали?

Финальная реплика:

> Вы не просто повторили приложение. Вы прошли путь от статической разметки до состояния, внешнего API, доступности и архитектуры. Теперь ваша задача в новом проекте — задавать те же вопросы: какие данные существуют, кто ими владеет, какие состояния увидит пользователь и где проходит граница ответственности.

- интернационализация;
- настоящие изображения с оптимизацией и lazy loading.

## Частые ошибки финальной интеграции

- одно значение состояния дублируется в нескольких компонентах;
- эффект используется для обычного вычисления;
- reset оставляет часть старых фильтров;
- API-объект проходит по всему приложению без адаптации;
- функция добавлена только для успешного сценария;
- компонент получает слишком много несвязанных обязанностей;
- визуальный polish скрывает проблемы клавиатуры или сети.

## Готово, если

- все основные сценарии работают совместно;
- состояние имеет понятных владельцев;
- избранное сохраняется;
- modal доступен с клавиатуры;
- сеть имеет loading, success, empty и error;
- приложение адаптивно;
- студент может объяснить поток данных, а не только показать результат.
